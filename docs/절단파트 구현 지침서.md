# 절단파트 구현 지침서 v3

> 최종 업데이트: 2026-06-03
> 관련 문서: [원자재_입고_공정_수정_지침서_v4.md](원자재_입고_공정_수정_지침서_v4.md)

이 문서는 절단파트 전체 워크플로우를 추적·완성하는 살아있는 문서다. 미정 항목은 명시하고, 확인되는 대로 갱신한다.

---

## 1. 파트별 역할

| 파트 | 메뉴 | 역할 |
|------|------|------|
| **사무실** | 프로젝트 → 호선/블록·블록강재리스트·블록BOM | 호선/블록 등록, 도면+잔재할당 등록, 일괄확정 |
| **자재파트** | 강재입고관리 | 정규 강재 입고/출고 (정규 흐름에서만 사용) |
| **현장** | 현장 작업일보 | 확정된 도면을 보고 판번호 선택 후 절단 시작/종료 기록 |
| **사무실/현장** | 잔재관리 | 등록잔재·현장잔재·여유원재 관리 + 돌발등록 |

## 2. 메뉴 구조

```
절단파트 (cutpart/)
├── 대시보드                  /cutpart
├── 프로젝트                  /cutpart/projects
│   ├── 호선/블록             ?tab=vessels
│   ├── 블록강재리스트         ?tab=list      (DrawingList)
│   └── 블록BOM리스트          ?tab=bom       (BOM)
├── 잔재관리                  /cutpart/scrap
│   ├── 잔재등록              (수동 등록 — 종류 라디오)
│   ├── 현장잔재 (REMNANT)
│   ├── 여유원재 (SURPLUS)    ← "현장원재" 라는 표현과 동의어
│   ├── 등록잔재 (REGISTERED) ← 블록강재등록 시 자동 생성 + 수동등록 둘 다
│   └── 돌발등록              (UrgentJob)

├── 작업일보 관리             /cutpart/worklog
└── 보고서                    /cutpart/reports
```

## 3. DB 모델 (핵심만)

### Project (호선/블록)
호선코드(예 RS01) + 블록코드(예 F50P) 단위. drawingLists 1:N.

### DrawingList (블록별 강재리스트, 1행 = 도면 1매)
| 컬럼 | 설명 |
|------|------|
| projectId | 소속 호선/블록 |
| drawingNo | 도면번호 |
| material/thickness/width/length | 규격 |
| block | 블록코드 |
| alternateVesselCode | 대체호선 — 본 호선 강재에 없을 때 다른 호선 강재로 매칭 |
| status | CAUTION / REGISTERED / WAITING / CUT |
| assignedRemnantId | 잔재 사용 시 Remnant FK (null이면 원재 사용) |
| heatNo | 절단완료 시 실사용 판번호 자동 기록 |

### SteelPlan (강재 1장 단위 — 정규 강재 입고)
원자재 지침서 v4 참고. 핵심: `reservedFor` = "호선/블록" 형식이면 선점됨.

### Remnant (잔재 — 3종류)
| 컬럼 | 설명 |
|------|------|
| type | REMNANT(현장잔재) / SURPLUS(여유원재) / REGISTERED(등록잔재) |
| shape | RECTANGLE / L_SHAPE / IRREGULAR |
| width1/length1/width2/length2 | 사이즈 |
| status | IN_STOCK / EXHAUSTED |
| reservedFor | "호선/블록" 형식 — 채워지면 선점됨 (확정) |
| sourceProjectId / sourceVesselName / sourceBlock | 발생 출처 |
| location | 보관 위치 |

**3상태 표시 (status + reservedFor 조합):**
- 재고: IN_STOCK + reservedFor=null
- 확정: IN_STOCK + reservedFor 있음 (호선/블록 표시)
- 소진: EXHAUSTED

### CuttingLog (절단 실적 — TBD 코드 재확인 필요)
작업일보에서 절단완료 시 기록되는 실적.

### UrgentJob (돌발작업 — TBD 코드 재확인 필요)
돌발등록 시 생성, 잔재와 직접 연결.

### SteelPlanHeat (판번호리스트 — 강재 1장 = 1행)
| 컬럼 | 설명 |
|------|------|
| vesselCode | 호선 |
| material/thickness/width/length | 규격 (SteelPlan과 FK 없이 느슨하게 연결) |
| heatNo | 판번호 (실물 철판에 새겨진 식별번호) |
| status | WAITING(대기) → CUT(절단완료) |

> SteelPlan 업로드 시 판번호 정보가 있으면 자동으로 함께 생성됨.

## 3-1. 판번호(HeatNo) 작업자 선택 흐름 ⭐ 중요

조선업 현장의 현실적 제약을 반영한 핵심 흐름.

### 현실적 제약
- 입고 시점엔 어느 호선·블록·도면에 어떤 판번호가 쓰일지 **모름**
- 현장 입출고가 빠르게 진행돼 판번호까지 일일이 체크 불가
- 입고 시 판번호리스트는 모두 WAITING(대기) 상태로 등록

### 작업자 선택 흐름
```
[현장] 작업자가 절단 작업 직전 실물 철판의 판번호 확인
   ↓
[전산] 현장작업일보 또는 작업일보관리에서 호선·블록·도면번호 선택
   ↓ 시스템: 동일 vesselCode + 동일 규격 + status=WAITING 판번호 목록 표시
[전산] 작업자가 실물 확인한 판번호 선택 → 절단완료 처리
   ├─ SteelPlanHeat: 선택된 판번호 WAITING → CUT
   ├─ DrawingList: 해당 도면 status=CUT + heatNo 컬럼에 사용판번호 기록
   ├─ SteelPlan: 사용된 1장 RECEIVED → COMPLETED
   │             + actualHeatNo / actualVesselCode / actualDrawingNo 기록
   └─ CuttingLog: 절단 실적 기록
```

### 판번호 dropdown 조회 조건
- 동일 vesselCode (대체호선 사용 시 alternateVesselCode)
- 동일 규격 (material / thickness / width / length)
- status = WAITING
- **확정(reservedFor)과 무관** — 자재파트가 어떤 판이 들어올지 모르므로 전체 표시

### 기록 위치 (절단완료 후 추적 가능)
| 위치 | 기록되는 정보 |
|------|---------------|
| DrawingList.heatNo | 그 도면에 사용된 판번호 |
| SteelPlan.actualHeatNo / actualVesselCode / actualDrawingNo | 그 강재 1장이 어디에 사용됐는지 |
| SteelPlanHeat.status | CUT (절단됨) |
| CuttingLog | 절단 실적 (작업자/일시/도면 등) |

### selectionPrintedAt = "선별지시서 출력일" = 선별지시일
- SteelPlan 모델의 별도 필드 (`selectionPrintedAt`)
- 일괄확정(reserve-bulk)과 무관 — `reservedFor` 채울 때 자동 안 찍힘
- 강재전체목록의 "🖨️ 선별지시서 출력" 버튼 누를 때만 그 시점이 selectionPrintedAt 으로 기록됨
- 트리거: `POST /api/steel-plan/mark-printed` (components/steel-plan-main.tsx → handlePrint)
- 운영 의도: 확정(블록 선점) 과 종이 인쇄 시점이 다를 수 있으므로 분리

### 절단 취소 시 복원
- DrawingList: CUT → WAITING + heatNo 초기화
- SteelPlanHeat: CUT → WAITING
- SteelPlan: COMPLETED → RECEIVED + actual* 필드 초기화
- CuttingLog: 삭제

> ⚠️ 현재 코드: cutting-logs/[id] PATCH·DELETE 가 docstring과 달리 SteelPlan 복원/COMPLETED 전환을 안 함 — Phase 1 수정 항목.

## 4. 정규 워크플로우 (정상 강재 사용)

```
[사무실] 프로젝트 → 호선 등록
    ↓ 호선코드(RS01) 생성

[자재파트] 강재입고관리
    ↓ 1. SteelPlan 엑셀 업로드 → REGISTERED
    ↓ 2. 실물 입고 처리 → SteelPlan: RECEIVED + DrawingList 자동 재계산
       (→ 자세한 내용은 원자재 지침서 v4 참고)

[사무실] 프로젝트 → 블록강재등록 모달
    ↓ 3. 도면 행 추가 (DrawingsMain upload 탭)
    │   ├─ 원재 사용 → assignedRemnantId=null
    │   └─ 잔재 사용 → assignedRemnantId 세팅 (셀렉터에서 reservedFor=null 잔재만 노출)
    ↓ 4. 초기 status = REGISTERED

[사무실] 블록강재리스트 → 일괄확정 버튼 (reserve-bulk POST)
    ├─ 원재 행: SteelPlan(RECEIVED, reservedFor=null) FIFO 선점
    │           → reservedFor="호선/블록", 실패 시 syncDrawingListBySpec 으로 CAUTION
    └─ 잔재 행: DrawingList.status = WAITING
                + Remnant.reservedFor = "호선/블록" (이미 다른 reservedFor 있으면 덮어쓰지 않음 — 선점 보호)

[현장] 현장 작업일보
    ↓ 5. 호선/블록/작업자 선택
    ↓    → 확정된(WAITING + reservedFor 매칭) 도면만 표시
    ↓ 6. 도면 선택 후 판번호 선택 (SteelPlanHeat WAITING)
    ↓ 7. 절단 시작 → 종료
    ↓    → DrawingList.status = CUT (heatNo 기록)
    ↓    → SteelPlan/SteelPlanHeat 업데이트 (원자재 지침서 v4 참고)
    ↓    → 발생 잔재 그 자리에서 Remnant(REGISTERED) 자동등록 가능

[사무실/현장] 잔재관리 → 잔재가 다시 [3]에서 재사용 가능
```

## 5. 돌발(Urgent) 워크플로우

**핵심**: 정규작업 아님. 갑자기 발생하는 비계획 절단업무. **정규 강재(SteelPlan)는 안 쓰고, 잔재만 사용**.

```
[돌발등록] 잔재관리 → 돌발등록 탭
    ↓ 사용할 자재 선택 (잔재관리 3종 중에서만)
    │   ├─ 등록잔재 (REGISTERED)
    │   ├─ 여유원재 (SURPLUS)
    │   └─ 현장잔재 (REMNANT)
    ↓ "어떤 돌발건 = 어떤 자재" 매핑 등록

[현장/사무실] 현장 작업일보 또는 작업일보관리
    ↓ 돌발건이 일보에 노출됨

[작업완료]
    ↓ CuttingLog 등 실적 기록
    ↓ 사용된 잔재는 EXHAUSTED 처리 (또는 잔여분 재등록)
```

**정규 흐름과의 차이**:
- BOM·확정·SteelPlan 단계 건너뜀
- 잔재 재고에서 즉시 자재 끌어와 작업
- DrawingList 와는 별개 트랙 (UrgentJob 모델)

**Why**: 조선업 특성상 도면에 없던 부재가 현장에서 갑자기 필요해지는 경우가 잦고, 정규 절차를 다 거치면 작업이 멈춤. 잔재 재고에서 바로 끌어쓰는 게 효율적.

## 6. 잔재 워크플로우 상세

### 발생 경로
1. **자동**: 정규 절단작업 후 남은 잔재 → 현장에서 Remnant(REGISTERED) 자동등록
2. **수동**: 잔재관리 → 잔재등록 탭에서 종류(REMNANT/SURPLUS/REGISTERED) 선택해 등록
3. **블록강재등록**: 사무실이 블록강재등록할 때 잔재를 도면에 할당 → 일괄확정 시 reservedFor 채워짐

### 확정(reservedFor) 메커니즘
- **목적**: 특정 블록에서 이 잔재를 쓰겠다고 선점 → 다른 블록이 중복 사용 못함
- **세팅**: reserve-bulk POST 가 assignedRemnantId 있는 도면 행 처리할 때
- **선점 보호**: 이미 다른 reservedFor 있는 잔재는 덮어쓰지 않음 (먼저 선점한 블록 우선)
- **해제**: reserve-bulk DELETE 가 본 프로젝트 호선/블록 매칭 reservedFor 만 클리어

### 사용 경로
- 정규 흐름: 블록강재등록 → 일괄확정 → 작업일보 → 절단
- 돌발 흐름: 돌발등록 → 작업일보 → 절단

### 3종 사용 패턴 (관찰)
- **현장잔재 (REMNANT)**: 정규 절단 후 남은 조각 — 가장 흔함
- **여유원재 (SURPLUS)**: 입고했지만 미사용한 판재 — "현장원재" 라고도 부름
- **등록잔재 (REGISTERED)**: 블록강재등록에서 사용되는 큰 단위 잔재

## 7. 상태 정의

### DrawingList.status
| 상태 | 의미 |
|------|------|
| CAUTION | SteelPlan에 해당 규격 자체 없음 (강재 등록조차 안 됨) |
| REGISTERED | SteelPlan에 있으나 미입고 |
| WAITING | 실물 입고됨 — 확정/절단 대기 |
| CUT | 절단 완료 |

### Remnant 표시상태 (status + reservedFor)
| 표시 | 조건 | 배지 |
|------|------|------|
| 재고 | IN_STOCK + reservedFor=null | 🟢 초록 |
| 확정 | IN_STOCK + reservedFor 있음 | 🔵 파랑 + "호선/블록" mono |
| 소진 | EXHAUSTED | ⚪ 회색 |

### SteelPlan.reservedFor
- 형식: `"호선/블록"` (예 "RS01/F50P")
- 구형 데이터: 블록코드만 — DELETE 시 호선코드 필터로 동명 블록 충돌 방지

## 9. 외부 강재 출고 로직 (선별지시 / 출고장) ⭐

> 위치: 강재입출고(`/cutpart/steel-plan`) + 외부출고관리(`/cutpart/external-shipout`)
> 상세 구현 문서: [shipout-implementation.md](shipout-implementation.md)
> 목적: 절단을 거치지 않고 **원판(SteelPlan)을 사외로 출고**한다. 절단 흐름(§4)과 별개 분기.

### 9.1 핵심 개념 — 선별지시 ≠ 출고장

명확히 다른 두 행위로 분리한다.

| 구분 | **선별지시 (Selection)** | **출고장 (Shipment)** |
|------|--------------------------|------------------------|
| 의미 | "이 강재들을 골라내라"는 준비/예약 지시 | 실제 사외 출고 (거래명세표 발행) |
| 강재 상태 (SteelPlan.status) | **입고(RECEIVED) 유지** | **외부(SHIPPED_OUT)** 전환 |
| 확정정보 표시 | 빨강 **"{호선} 선별"** 또는 **"{호선/블록} 선별"** | 빨강 **"{호선} 출고"** 또는 **"{호선/블록} 출고"** |
| 판번호 (SteelPlanHeat) | 변화 없음 (WAITING 유지) | **외부(SHIPPED)** 전환 |
| 외부출고관리 등록 | 없음 | **출고장 목록에 등록** |
| 되돌리기 | 선별 취소 (확정정보 배지 클릭) | 출고 취소 (RECEIVED + 판번호 WAITING 복원) |
| 산출물 | **선별지시서** (창고 픽업용) | **거래명세표(출고증)** |

순서: **선별지시(준비) → 출고장(실제 출고)**. 선별만 하고 출고장은 나중에 만들 수 있고, 선별 없이 바로 출고장도 가능.

### 9.2 상태 / 확정정보 / 판번호 규칙

- **확정정보 컬럼 우선순위**:
  1. `status = SHIPPED_OUT` → 빨강 "{라벨} 출고"
  2. 선별 마킹 있음 → 빨강 "{라벨} 선별"
  3. 블록확정(`reservedFor`, 절단용) → 보라 "{호선/블록}"
  4. 없음 → "-"
- {라벨} = 호선 또는 호선/블록 (강재매칭 경로는 매칭이름도 가능).
- 선별/출고 마킹은 **블록확정(reservedFor)·내부출고(ISSUED) 가드를 건드리지 않는다** (별도 필드 `shipoutMarkedAt`/`shipoutLabel` 사용).
- 출고장이 실제 생성될 때만 status/판번호가 외부로 바뀐다 (유령 출고 방지 — `POST /api/shipments` 트랜잭션에서만).

### 9.3 현재 구현 현황 — 출고 관련 진입점 (정리 대상)

| # | 진입점 | 위치 | 입력 | 현재 동작 | 불일치/중복 |
|---|--------|------|------|-----------|-------------|
| 1 | **외부출고리스트** | 헤더 버튼 | 엑셀(출고예정) | 자동매칭→출고 카트→출고장 마법사→외부 | 출고장 전용 |

### 9.8 절단 ↔ 출고 상호배제 규칙 ★ (데이터 정합성 핵심)

> 전제: **절단 확정정보 = `reservedFor`**, **출고 확정정보 = `shipoutMarkedAt`**. 두 마킹 모두 status를 RECEIVED로 유지 → status 화이트리스트만으로는 상호배제 불가. **반대편 필드를 where에서 명시적으로 제외**해야 한다.

**규칙 매트릭스**
1. **절단 풀**(가용수량·도면 sync·블록확정 reserve/reserve-bulk·절단완료 폴백·일괄투입 issue-bulk)은 `shipoutMarkedAt: null` 추가 → **출고예정 강재 제외**.
2. **출고 풀**(shipout-match·shipout-mark·shipments POST·excel-upload·강재매칭 selectable)은 `reservedFor: null` 추가 → **블록확정(절단용) 강재 제외**.
3. **판번호 풀**(heat-options·절단완료 WAITING→CUT)은 대응 SteelPlan이 `shipoutMarkedAt != null`이면 제외 (마킹 전파 또는 join).
4. **쓰기 경로**(issue-bulk·shipout-mark·shipments POST)는 반대편 마킹이 있으면 **409 차단** (UI 우회·경쟁상태 대비, 양방향 대칭).
5. **정식 전이**(출고예정→투입/출고)는 반대편 마킹을 **명시적으로 해제 후** 진행. issue-bulk가 출고마킹을 조용히 삭제하는 현재 동작은 차단(409)로 대체.
6. status(ISSUED/COMPLETED/SHIPPED_OUT) 제외는 현행 RECEIVED 화이트리스트 유지(부수효과)하되, 위 명시 제외를 더한다.

### 9.10 원판 + 잔재 통합 출고 ⭐ (2026-06-20, 커밋 adb9e77f)

선별 목록(출고 예약 풀)에 **잔재(Remnant)** 를 담아 **원판(SteelPlan)과 같은 출고장으로 함께 외부 출고**한다. 원판 기존 출고 기능은 무손상(전부 추가 분기).

**진입점**: 강재입출고 → [선별 목록] 탭 → **[잔재 추가]** 버튼([선별 취소] 우측) → 모달에서 타입(여유원재 SURPLUS / 등록잔재 REGISTERED / 현장잔재 REMNANT) 선택 → 가용 잔재 목록에서 선택 → 선별 목록에 추가(마킹) → 선별 목록에서 원판과 함께 선택 → 출고 카트 → 출고장 마법사 → 거래명세표.

- `ShipmentItem.steelPlanId` → **nullable**, `remnantId String?` + FK(Restrict) 추가. 출고 항목은 **`steelPlanId` XOR `remnantId`** (정확히 한쪽만, POST에서 강제).
- `Remnant.shipoutMarkedAt DateTime?` 추가 — 선별 목록 멤버십 마킹(원판의 `shipoutMarkedAt`과 동일 모델). + `Remnant.shipmentItems` 역관계.

**상태 lifecycle (잔재)**:
- 선별 추가 = `shipoutMarkedAt` 마킹 (status는 IN_STOCK 유지, 되돌리기 가능).
- 출고 확정 = `status: IN_STOCK → EXHAUSTED` + `shipoutMarkedAt` 정리 (원자적 `updateMany where status:IN_STOCK`, race 시 한쪽만 성공).
- 출고 취소 = `status: EXHAUSTED → IN_STOCK` 복원.

**불변식 (깨면 자재 이중사용/공정 정합성 붕괴)**:
1. 출고되는 잔재는 **항상 `status=IN_STOCK` + `reservedFor=null`**. PENDING(미절단)·절단확정(reservedFor)·EXHAUSTED 잔재는 출고 차단. → 마킹(remnants/shipout)·서버 POST·추가 모달 3곳에서 강제. (출고원천을 IN_STOCK으로 고정해야 취소 복원이 정합)
2. **절단↔출고 상호배제**가 원판과 동일하게 잔재에도 적용 (§9.8). 잔재 `reservedFor`는 `drawings/reserve-bulk`가 채움 → 출고 경로에서 제외.

**구현 파일**:
- `app/api/remnants/shipout/route.ts` (신규) — GET(선별된 잔재 목록), POST mark/unmark. mark는 `where status:IN_STOCK, reservedFor:null, shipoutMarkedAt:null`.
- `components/selection-list-tab.tsx` — 원판(`/api/steel-plan?shipoutMarked=true`) + 잔재(`/api/remnants/shipout`) 병합 표시, "구분(원판/잔재)" 컬럼, [잔재 추가] 모달(`RemnantPickerModal`, `?type=&onlyAvailable=true`), 카트담기/선별취소 kind별 분기.
- `components/shipout-cart.tsx` — `ShipoutCartItem.kind`("plate"|"remnant", 없으면 plate) + `remnantId`. 카트 고유키는 `steelPlanId` 필드 재사용(잔재는 Remnant.id 저장 — cuid라 충돌 없음).
- `components/shipout-bar.tsx` — 잔재는 판번호 매칭(heat-match) 제외, 판번호 중복검사 제외, payload를 kind별 분기(remnant=remnantId, plate=steelPlanId/heatId), Step1에 "잔재" 배지.
- `app/api/shipments/route.ts` — 항목 XOR 검증, 원판/잔재 분리 검증(중복·이미출고·소진·reservedFor), tx에서 잔재는 ShipmentItem(remnantId·steelPlanId null·heatNo 스냅샷만, SteelPlanHeat 미생성) 생성 + Remnant EXHAUSTED. 원판 경로 불변.
- `app/api/shipments/[id]/cancel/route.ts` — `item.remnantId`면 잔재 IN_STOCK 복원, 아니면 기존 원판 복원.
- `components/invoice-print.tsx` — 불규칙(IRREGULAR)·치수 0 잔재는 폭/길이/면적 칸 공란(원판은 영향 없음).

**적대적 검증** (workflow, 24 agents): 원판 무손상 확인 + 확정 결함 6종 수정 — [HIGH] 잔재 reservedFor 가드 누락, [MED] PENDING 잔재 출고/취소 승격→IN_STOCK 한정, [MED] 양쪽 FK 빈 유령행→XOR 검증, [LOW] 소진 TOCTOU→원자화, [LOW] 0×0 명세표, [LOW] vesselCode "-" 저장.

> **거래명세표 호환**: ShipmentItem은 출고 시점 사양 스냅샷(vesselCode/material/치수/중량/heatNo/block)을 보관하므로, 명세표·출고상세는 잔재 행도 `steelPlanId` 참조 없이 스냅샷만으로 정상 렌더(코드 변경 거의 없음).
